# Joy-Con IMU 处理方案对比

本文对比原 `joyconrobotics` 实现和当前 `joyrebot_teleop` 使用的 IMU 姿态估计方案。

## 原实现

原实现对 roll、pitch 和 yaw 采用了两套不同的计算方式：

```text
加速度计 + 单步陀螺仪
        → 欧拉角互补滤波
        → roll / pitch

陀螺仪
        → 旋转三个方向向量
        → direction_X[1]
        → yaw
```

主要特点：

- 每次更新都会先把 roll、pitch 清零，再进行一次陀螺仪计算和加速度计融合，没有连续
  保存完整三维姿态。
- yaw 使用旋转后 X 方向向量的 Y 分量 `direction_X[1]` 近似，并不是真正的 yaw 角。
- roll、pitch、yaw 最后还会乘以约 `2.09` 的经验放大系数。
- 每次只使用 HID 报告中的第一个 IMU 样本。
- 陀螺仪从度每秒转换到弧度每秒时少乘了一个系数 2。

因此大角度转动时容易出现 yaw 非线性、不同旋转轴互相耦合以及姿态变化过于敏感。

## 当前实现

当前方案使用六轴 Mahony 四元数滤波：

```text
三轴陀螺仪
        → 正确转换为 rad/s
        → 四元数连续积分
                     ┐
三轴加速度计 → 重力方向误差 ─→ Mahony 反馈校正
                     ↓
           roll / pitch / continuous yaw
```

主要改动：

- 使用一个完整四元数连续保存和更新三维姿态。
- yaw 从完整四元数计算并进行连续展开，不再使用 `direction_X[1]`。
- 加速度计只负责修正重力可观测的 roll、pitch，不把它当作 yaw 航向参考。
- 每个约 15 ms 的 HID 报告包含三个 IMU 样本，三个样本都会且只会融合一次。
- 快速平移导致加速度模长明显偏离 1 g 时，暂时降低或关闭重力校正，避免把平移误判为
  手柄倾斜。
- 移除了原来的 `π/1.5` 姿态放大，输出直接使用真实弧度。
- 静止标定会清除陀螺仪零偏并重新建立姿态零点。

## 输出是否为原始 yaw

Joy-Con 没有直接输出 yaw，只输出三轴角速度和三轴加速度。因此无论新旧方案，
`/joycon_input/pose` 中的姿态都不是原始 yaw。

当前数据链路为：

```text
Joy-Con 原始 IMU
→ Mahony 四元数滤波
→ 处理后的 roll/pitch/yaw
→ 转换为 geometry_msgs/PoseStamped 中的四元数
→ /joycon_input/pose
```

`joycon_input_node` 终端面板显示的 yaw 也是上述滤波和清零处理之后的 yaw。

## 仍然存在的物理限制

Joy-Con 没有磁力计，重力方向不能提供水平航向信息。因此当前 yaw 虽然比原算法正确、
连续且轴间耦合更小，但本质上仍由陀螺仪积分得到，长时间运行可能因零偏缓慢漂移。
出现明显漂移时，应将手柄水平静置并重新执行标定。
